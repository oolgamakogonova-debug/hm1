# Marketplace Architecture (C4 + Docker)

## Что сделано
- Спроектирована архитектура маркетплейса (контейнерный уровень)
- Построена C4 Container диаграмма
- Поднят один сервис в Docker с health-check endpoint `/health` (200 OK)
- Описаны домены, зоны ответственности, разбиение по сервисам и границы владения данными

## Архитектура (коротко)
Marketplace состоит из основных доменов:
- Users (покупатели/продавцы)
- Catalog (товары)
- Feed/Recommendations (персонализированная лента)
- Orders (оформление заказов)
- Payments (учет платежей + интеграция с провайдером)
- Notifications (уведомления о статусах)

Вход в систему через API Gateway. Интеграции с внешними системами: Payment Provider и Email/SMS/Push Provider.

## Домены и ответственность (DDD)

### Users
**Отвечает за:** регистрацию/аутентификацию, роли (buyer/seller), профиль, контакты, адреса.  
**Не отвечает за:** товары, заказы, платежи, уведомления.

### Catalog
**Отвечает за:** товары, категории, атрибуты, цены, (опционально) остатки/наличие — **источник истины** по карточкам товара.  
**Не отвечает за:** оплату и жизненный цикл заказа.

### Feed/Recommendations
**Отвечает за:** персонализированную выдачу (ранжирование, правила, сигналы), сбор кандидатов из каталога/поиска.  
**Не отвечает за:** хранение “истинных” данных о товарах и заказах (использует данные других доменов).

### Orders
**Отвечает за:** оформление заказа, статусы заказа, историю изменений, состав заказа (позиции).  
**Не отвечает за:** проведение платежа (только инициирует и реагирует на результат).

### Payments
**Отвечает за:** учет платежей, создание/проведение платежа через провайдера, статусы платежа, ретраи/идемпотентность.  
**Не отвечает за:** владение статусом заказа (только сообщает результат Order’у).

### Notifications
**Отвечает за:** отправку уведомлений о статусах (email/sms/push), шаблоны, историю отправок.  
**Не отвечает за:** бизнес-решения (кому/когда менять статус) — только доставка.

## Распределение доменов по сервисам и логика разбиения

### Домены → сервисы
- **Users domain → Users Service**
- **Catalog domain → Catalog Service**
- **Feed domain → Feed/Recommendation Service**
- **Orders domain → Order Service**
- **Payments domain → Payment Service**
- **Notifications domain → Notification Service**

### Почему так (логика разбиения)
- **Слабая связанность:** заказы не “влезают” в платежную логику и наоборот.
- **Независимое масштабирование:** Catalog/Feed могут быть самыми нагруженными, Payments — критичен по надежности.
- **Упрощение развития:** домены меняются независимо, проще релизы и поддержка.
- **Надежность:** уведомления/платежи можно обрабатывать асинхронно и не блокировать пользователя.

## Границы владения данными (data ownership)

Принцип: **у каждого сервиса своё хранилище, общих БД нет** (нет “разделяемой базы данных”).

- **Users Service владеет данными:** пользователи, роли, профиль, адреса (Users DB)
- **Catalog Service владеет данными:** товары, категории, атрибуты, цены (Catalog DB)
- **Feed Service владеет данными:** фичи/сигналы/кэш ранжирования (Feed store/Redis)
- **Order Service владеет данными:** заказы, позиции, статусы, история (Order DB)
- **Payment Service владеет данными:** платежи, транзакции, статусы (Payment DB)
- **Notification Service владеет данными:** шаблоны/логи отправок/статусы доставки (Notification DB/лог)

## Взаимодействия сервисов (sync/async)

### Синхронные (REST через API Gateway)
- Клиент → API Gateway → Users/Catalog/Feed/Orders/Payments
- Feed может синхронно получать кандидатов из Catalog/Search (упрощённо)

### Асинхронные (события/очередь)
- **Order Service → Notification Service:** `OrderStatusChanged` (уведомить пользователя)
- **Order Service → Payment Service:** `OrderCreated` (инициировать оплату)
- **Payment Service → Order Service:** `PaymentSucceeded` / `PaymentFailed` (обновить статус заказа)
- **Catalog Service → Feed Service:** `ProductUpdated` (обновить кэш/выдачу)

**Почему async:** чтобы платежи/уведомления не блокировали пользовательский запрос и система оставалась устойчивой (eventual consistency допустима для связанных доменов).
