## Альтернативные варианты декомпозиции и trade-offs (8–10 баллов)

Ниже рассмотрены 3 варианта разбиения. В качестве **финального** выбран вариант C (domain-based microservices), т.к. он лучше соответствует требованиям маркетплейса (каталог/заказы/платежи/уведомления/персонализация) и будущему росту.

---

### Вариант A — Монолит (Modular Monolith)
**Идея:** один backend-сервис (один деплой), внутри модули: Users, Catalog, Feed, Orders, Payments, Notifications. Единая БД (одна схема), слои/модули разделены кодом.

**Плюсы:**
- Самая простая разработка/дебаг/деплой.
- Нет сетевых вызовов между доменами → меньше инфраструктуры.
- Проще транзакции (одна БД).

**Минусы:**
- Тяжело независимо масштабировать горячие части (Feed/Catalog).
- Растёт связанность кода и риск “большого комка”.
- Сложнее параллельная работа команд и независимые релизы.
- Со временем миграция в микросервисы становится болезненной.

**Когда подходит:** маленькая команда, MVP, низкая нагрузка, нет требований к независимому масштабированию.

---

### Вариант B — Сервис по слоям (Layered / Technical decomposition)
**Идея:** разбиение не по доменам, а по техническим слоям:  
- API Service (всё HTTP)  
- Core Service (вся бизнес-логика)  
- Data Service (все операции с БД)  
или например отдельные сервисы: Auth Service, CRUD Service, Recommendation Service.

**Плюсы:**
- Частично разделяет ответственность по технологиям.
- Можно отдельно тюнить “тяжёлую” ML-часть (Feed).

**Минусы:**
- Доменные границы размыты: Orders/Payments/Users начинают “переезжать” между слоями.
- Высокая связанность между сервисами по вызовам (чаттеринг), сложнее сопровождение.
- Data Service часто превращается в “общую БД через API” → нарушается data ownership.
- Трудно объяснить владение данными и ответственность доменов.

**Когда подходит:** редко; обычно как промежуточный этап или в очень специфичных командах, где строго делят по технологиям.

---

### Вариант C — Доменная декомпозиция (Domain-based microservices) — **выбранный**
**Идея:** сервисы соответствуют доменам: Users, Catalog, Feed, Orders, Payments, Notifications (+ API Gateway).  
Каждый сервис владеет своими данными (database-per-service). Между доменами: sync REST + async события.

**Плюсы:**
- Чёткая ответственность и границы (легко проверять по требованиям).
- Независимое масштабирование (Feed/Catalog отдельно от Payments).
- Командная разработка и независимые релизы.
- Устойчивость: async для платежей/уведомлений снижает блокировки.
- Ясная модель владения данными (нет shared DB).

**Минусы:**
- Больше инфраструктуры (gateway, брокер/очередь, наблюдаемость).
- Появляется eventual consistency между доменами (нужно учитывать).
- Нужны идемпотентность и ретраи обработчиков событий.

**Когда подходит:** маркетплейс с ростом нагрузки, несколькими доменами и интеграциями (платежи/уведомления), где важны масштабируемость и независимость частей.

---

## Итоговый выбор и обоснование

**Выбран вариант C (domain-based microservices)**, потому что он:
- напрямую отражает требования кейса (каталог, пользователи, заказы, платежи, уведомления, персонализация);
- позволяет независимо масштабировать самые нагруженные части (Feed/Catalog) и изолировать критичный Payments;
- обеспечивает понятные границы владения данными (нет разделяемых БД между сервисами);
- позволяет вынести долгие операции (платежи/уведомления) в асинхронные процессы, чтобы не блокировать пользователя.

При этом на текущем этапе бизнес-логика не реализуется — мы фиксируем архитектурный каркас, контейнерную диаграмму и демонстрируем запуск одного сервиса в Docker.
